# PLF-032: Add backlog planning contract builder

## Objective

Add a command that analyzes selected backlog tickets with separate read-only child Pi contexts and produces a single backlog planning contract for human approval.

## Context

The user wants to select several backlog tickets and ask Pi to prepare them for planning. The analysis must not happen as one large global LLM context. Each requested ticket, discovered dependency ticket, and proposed split should be analyzed in a focused child context where practical. The parent orchestrator should aggregate those results into one contract that says exactly what will be edited, created, split, and moved if the human approves.

This is not a dry-run followed by a second strategic run. This command performs the strategic analysis once and writes the proposed contract that later execution should apply.

## Scope

In scope:

- Add a command such as `/ticket-backlog-contract PLF-001 PLF-002`.
- Require selected tickets to exist in `tickets/backlog/`.
- Run `/ticket-status`-equivalent and `/ticket-doctor`-equivalent deterministic checks before analysis.
- Launch bounded read-only child Pi analysis for each selected ticket.
- Discover explicit dependencies and analyze dependency tickets in separate read-only child contexts when they are in backlog or otherwise relevant.
- Ask child analyses to identify missing information, dependency blockers, proposed ticket edits, and split recommendations.
- Aggregate child outputs into one proposed backlog planning contract artifact.
- Include every proposed ticket edit, new ticket, dependency update, split, and move in the contract.
- Include source ticket fingerprints so later execution can detect staleness.
- Hand the contract summary back to the parent LLM for human review.

Out of scope:

- Editing tickets.
- Creating split tickets.
- Moving tickets to `planned/`.
- Applying human approval.
- Implementing planned tickets.
- Re-running strategic analysis during apply.

## Acceptance Criteria

- [ ] The command accepts one or more backlog ticket IDs.
- [ ] Each requested ticket is analyzed in a separate bounded read-only child context.
- [ ] Dependency tickets are identified and either analyzed separately or reported as unresolved with a clear reason.
- [ ] The resulting contract lists all proposed edits, splits, new tickets, dependency changes, and ticket moves.
- [ ] The contract clearly distinguishes simple requested moves from additional changes requiring human attention.
- [ ] The contract contains source fingerprints for staleness checks.
- [ ] The command does not mutate ticket files or ticket directories.
- [ ] The command reports a clear approval prompt or next action for the human.
