# PLF-019: Define advisory artifact conventions for ticket commands

## Objective

Define the minimal storage layout, metadata, and helper behavior for advisory artifacts produced by future ticket workflow commands.

## Context

The planned command system should use child Pi sessions mostly as read-only advisors. Their outputs must be durable and easy to inspect, but they must not replace ticket files or ticket directories as the source of truth.

A shared convention is needed before adding readiness, implementation-plan, or verification commands so each command does not invent its own artifact format.

## Scope

In scope:

- Define an advisory artifact location, likely under `tickets/.artifacts/`.
- Add subdirectories for readiness briefs, implementation plans, verification briefs, and possibly completion briefs.
- Document that artifacts are advisory only.
- Add helper logic for deterministic artifact paths.
- Add helper logic to compute or record enough ticket fingerprint information to detect stale artifacts.
- Include generated timestamp, command name, ticket ID, ticket path, ticket state, and advisory notice in artifact metadata where practical.

Out of scope:

- Generating real readiness, implementation-plan, or verification content.
- Child Pi process orchestration.
- Ticket state transitions.
- Commit automation.
- Treating artifacts as authoritative workflow state.

## Acceptance Criteria

- [ ] The chosen artifact layout is documented.
- [ ] Artifact paths are deterministic for a ticket ID and artifact type.
- [ ] Artifact metadata includes enough information to identify the source ticket and detect likely staleness.
- [ ] Documentation clearly states that ticket files and their directories remain authoritative.
- [ ] The implementation does not repurpose ticket state directories for artifacts.

## Dependencies

- `PLF-018` — the ticket workflow extension foundation should exist first.

## Implementation Notes

Likely files:

- `.pi/extensions/ticket-workflow/`
- `tickets/.artifacts/README.md`
- `tickets/.artifacts/readiness/.gitkeep`
- `tickets/.artifacts/plans/.gitkeep`
- `tickets/.artifacts/verification/.gitkeep`
- `tickets/.artifacts/completion/.gitkeep` if included
- `docs/ticket-workflow-commands.md`

Keep metadata small and inspectable. A content hash of the source ticket is preferable to relying only on modification time.

## Verification

- Inspect the artifact documentation manually.
- Run existing ticket workflow command(s), especially `/ticket-status`, to ensure no regression.
- If helper output is exposed through a dry-run command or tests, confirm generated paths and hashes are stable.
