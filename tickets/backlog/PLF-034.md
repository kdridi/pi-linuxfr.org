# PLF-034: Add planned implementation contract builder

## Objective

Add a command that analyzes all planned tickets with separate read-only child Pi contexts and produces one implementation execution contract for human approval.

## Context

The user wants to spend focused time reviewing a full implementation plan for the day's planned work, then approve a long-running automation pass. The analysis should not be one large global LLM pass. Each planned ticket should be planned in its own child context, and the parent orchestrator should aggregate the per-ticket plans into a global execution contract.

This command performs the planning analysis once. Later trusted execution must consume the approved contract rather than redoing the strategic planning.

## Scope

In scope:

- Add a command such as `/ticket-implementation-contract`.
- Inspect all `tickets/planned/PLF-*.md` tickets, or a bounded selected subset if the implementation needs an explicit limit.
- Run deterministic status and doctor checks before analysis.
- Order tickets according to explicit dependencies where possible.
- Refuse or stop with a clear report when dependencies are unresolved, cyclic, missing, or outside the planned/completed set in a way that prevents execution.
- Launch a separate bounded read-only child Pi planning analysis for each planned ticket.
- Ask each child to produce objective, scope, likely files, implementation steps, verification commands, risks, stop conditions, and suggested commit message.
- Aggregate per-ticket plans into one implementation execution contract.
- Include source fingerprints for planned tickets and relevant existing advisory artifacts.
- Hand the global contract summary back to the parent LLM for human review.

Out of scope:

- Moving tickets to `ongoing/`.
- Editing code.
- Running tests.
- Verifying implementation results.
- Committing.
- Re-planning during execution.

## Acceptance Criteria

- [ ] The command analyzes planned tickets in separate child contexts.
- [ ] The command produces a global ordered execution contract.
- [ ] The contract includes per-ticket implementation plans and verification commands.
- [ ] The contract includes source fingerprints for staleness checks.
- [ ] The contract clearly reports dependency ordering, blockers, and stop conditions.
- [ ] The command does not mutate ticket files or project implementation files.
- [ ] The output is suitable for a human to approve a long-running trusted implementation run.
