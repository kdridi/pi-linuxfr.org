# PLF-016: Clean HTML snippets in candidate extraction output

## Objective

Improve snippets returned by `linuxfr_extract_candidates` for HTML seed sources so they show readable human context instead of raw structural fragments.

## Context

`PLF-001` added bounded local candidate extraction. Manual post-reload testing confirmed the tool works, but snippets extracted from HTML seed pages can include partial markup-adjacent text such as schema attributes, score labels, or truncated structural fragments. This makes candidate output less useful for LLM and human review.

This is a cleanup ticket, not a scope expansion: extraction must remain local-only and bounded.

## Reproduction

Run the tool against an existing local tag page:

```json
{
  "sourcePaths": [
    "data/raw/pages/feed-a554f7e7d2.atom",
    "data/raw/pages/tag-07e4f4520b.html"
  ],
  "query": "agent llm",
  "limit": 5
}
```

Observed result excerpt for the DarkMoon candidate:

```text
- score 20 news: https://linuxfr.org/news/darkmoon-un-moteur-libre-de-pentest-autonome-avec-agents-ia-mcp-et-outillage-conteneurise — DarkMoon : un moteur libre de pentest autonome avec agents IA, MCP et outillage conteneurisé
  source: data/raw/pages/tag-07e4f4520b.html
  reason: classified as news; matched agent; has title or anchor text
  matched: agent
  excerpt: score2 news" itemscope itemtype="http://schema.org/Article"> Dark
```

Problem: the excerpt contains structural HTML-adjacent text (`score2 news" itemscope itemtype=...`) instead of useful local context around the candidate.

Desired behavior: the excerpt should be readable plain text from the candidate block, for example something closer to the article title, type, score, date, tags, or summary, without raw attributes or broken fragments.

## Scope

In scope:

- Improve snippet cleanup for HTML seed sources used by `linuxfr_extract_candidates`.
- Strip raw tag attributes and schema fragments from snippets.
- Prefer candidate-local readable text when available.
- Keep snippets short and deterministic.
- Add unit-level checks for snippet cleanup helpers or extraction behavior.

Out of scope:

- Crawling or fetching new pages.
- Ranking changes unrelated to snippet cleanliness.
- Full HTML semantic parsing if a simpler deterministic cleanup is enough.
- Changing wiki generation.

## Acceptance Criteria

- [ ] The reproduction above no longer returns snippets containing raw `itemscope`, `itemtype`, or similar broken attribute fragments.
- [ ] HTML snippets remain plain text and compact.
- [ ] Atom feed snippets still work at least as well as before.
- [ ] The implementation stays local-only and bounded.
- [ ] Unit or focused runtime scenarios cover the cleanup cases below.

## Unit Test Scenarios

At minimum, cover these focused scenarios with helper-level tests or a small local fixture-based runtime check:

1. **HTML article fragment with schema attributes**
   - Input contains an `<article class="score2 news" itemscope itemtype="http://schema.org/Article">` block and an `<a>` title.
   - Expected snippet does not include `itemscope`, `itemtype`, `schema.org`, or raw attribute text.
   - Expected snippet includes readable candidate-local text such as the title.

2. **Partial HTML around an anchor**
   - Input context starts in the middle of an HTML tag or attribute because the anchor is near the snippet boundary.
   - Expected cleanup drops the partial tag/attribute fragment rather than exposing it.

3. **HTML entity decoding**
   - Input contains entities such as `&nbsp;`, `&amp;`, `&#x202F;`, or accented text.
   - Expected snippet is decoded and whitespace-compacted.

4. **Atom snippet regression**
   - Input is a small Atom `<entry>` with `<content type="html">`.
   - Expected snippet remains readable and does not lose the entry title or content summary.
