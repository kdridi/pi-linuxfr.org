# PLF-004: Add transparent candidate ranking

Backlog tickets may be incomplete. Before moving this ticket to `planned`, refine the scope and verification method.

## Objective

Rank extracted LinuxFr candidate detail URLs with simple transparent scores and reasons so agents can choose what to collect next without improvising.

## Context

The expanded radar review found that candidate selection was manually improvised and not reproducible. The V2 roadmap places candidate ranking immediately after candidate extraction.

Motivating sources:

- `docs/linuxfr-v2-system-roadmap.md`
- `docs/prototype-prompts/agentic-ai-career-radar-expanded/review.md`
- `tickets/planned/PLF-001.md`

## Scope

In scope:

- Score already extracted local candidates by simple factors such as matched terms, source type, title/anchor text, tag/context match, date when available, and source seed context.
- Return a shortlist with score components or human-readable reasons.
- Preserve rejected/noisy candidates when useful for auditability.

Out of scope:

- LLM semantic ranking as the only ranking mechanism.
- Automatic detail collection.
- Full relevance modeling or vector search.
- Report generation.

## Acceptance Criteria

- [ ] To be refined before planning.
- [ ] Candidate scores are inspectable and explainable.
- [ ] The ranking operates on local candidate data only.
- [ ] The output helps a human or agent decide which explicit URLs to collect next.

## Dependencies

- `PLF-001` — candidate ranking needs candidate extraction output or an equivalent structured candidate table.

## Implementation Notes

Keep the first scoring heuristic deliberately boring. The value is repeatability and transparency, not perfect ranking.
