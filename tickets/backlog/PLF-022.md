# PLF-022: Add planned ticket implementation plan command

## Objective

Add a read-only advisory command that produces an implementation plan for a ticket in `tickets/planned/`.

## Context

Planned tickets should be implementable by a fresh Pi session. A dedicated implementation-plan command can prepare the work in a separate context by reading the ticket, relevant docs, and likely code paths, then producing a durable plan for the orchestrator and human to review before activation.

## Scope

In scope:

- Add a command such as `/ticket-plan <ticket-id>`.
- Require the target ticket to exist in `tickets/planned/`.
- Check dependencies enough to warn about unresolved items.
- Use read-only child Pi analysis.
- Write an advisory implementation plan to the agreed artifact location.
- Include objective restatement, scope boundaries, likely files, step-by-step implementation approach, verification plan, risks, and stop conditions.
- Map acceptance criteria to verification steps where practical.
- Hand the advisory plan back to the parent LLM according to the `PLF-029` convention.

Out of scope:

- Activating the ticket.
- Editing code.
- Editing ticket scope.
- Resolving dependencies automatically.
- Running a write-capable implementation child.

## Acceptance Criteria

- [ ] The command refuses tickets that are not currently in `tickets/planned/`.
- [ ] The generated plan includes an advisory notice and source ticket fingerprint.
- [ ] The plan identifies likely files and explicitly marks uncertainty where present.
- [ ] The plan maps ticket acceptance criteria to implementation and verification steps.
- [ ] The plan includes stop conditions for scope drift or missing information.
- [ ] The command hands the plan back to the parent LLM or prints the suggested handoff in print mode.
- [ ] The command does not activate or modify the ticket.

## Dependencies

- `PLF-019` — shared advisory artifact conventions.
- `PLF-020` — read-only child Pi runner.
- `PLF-029` — parent LLM handoff convention for advisory command results.

## Implementation Notes

Likely files:

- `.pi/extensions/ticket-workflow/`
- `.pi/extensions/ticket-workflow/prompts/plan.md` or equivalent inline prompt helper
- `tickets/.artifacts/plans/`
- `docs/ticket-workflow-commands.md`

This command should make the later `planned/` -> `ongoing/` decision easier, but should not perform that transition.

## Verification

- Run the command on a controlled planned ticket.
- Inspect the generated implementation plan for concrete steps and non-goals.
- Confirm stale plan detection works after the ticket changes.
- Confirm no ticket files move or change.
