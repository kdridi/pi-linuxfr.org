# PLF-015: Refine the initial backlog into planned candidates

Backlog tickets may be incomplete. Before moving this ticket to `planned`, decide whether backlog refinement itself should be ticketed or handled manually.

## Objective

Review the newly created backlog tickets, identify duplicates or oversized items, and propose the next small set of tickets that should be refined into `planned`.

## Context

After commit `b16fa3b`, the project intentionally captured V2 capabilities, prototype lessons, workflow debts, and experimental use cases into `tickets/backlog/` before implementing more code. That inbox will need a follow-up refinement pass so it does not become an idea graveyard.

Motivating sources:

- `tickets/README.md`
- `docs/linuxfr-v2-system-roadmap.md`
- `docs/prototype-prompts/agentic-ai-career-radar-expanded/review.md`
- `docs/use-case-challenge-prompts/`

## Scope

In scope:

- Inspect all backlog tickets.
- Group them by implementation capability, research experiment, documentation, and exploratory use case.
- Identify duplicate, dependent, premature, or oversized tickets.
- Recommend which 1 to 3 tickets should be refined into `planned` next.

Out of scope:

- Implementing any feature.
- Moving tickets without user validation.
- Rejecting tickets without an explicit reason and replacement when applicable.
- Creating a project-management index or automation.

## Acceptance Criteria

- [ ] To be refined before planning.
- [ ] The refinement pass produces a concise recommendation for next planned tickets.
- [ ] It respects the one-active-ticket workflow and does not move anything to `ongoing`.
- [ ] It keeps backlog tickets as useful historical records.
