# PLF-030: Add experimental trusted ticket autopilot command

## Objective

Add an experimental command that can drive one ticket through the local ticket workflow with trusted automation, using the existing advisory commands as checkpoints and stopping when checks fail or scope becomes uncertain.

## Context

The current ticket workflow is intentionally manual and advisory: commands inspect tickets, produce plans or recommendations, and ask for explicit human approval before transitions. After the workflow doctor exists, it should be possible to experiment with a higher-trust orchestrator for users who explicitly accept the risk of automation.

The desired model is an initial Pi orchestration layer that can run existing ticket workflow commands, interpret their advisory output, perform approved workflow transitions, launch implementation work, run verification, and commit when all required checks pass. Some existing commands may themselves launch read-only child Pi sessions; the autopilot must preserve those safety boundaries and avoid unbounded nested automation.

## Scope

In scope:

- Add an explicitly experimental command such as `/ticket-autopilot <ticket-id>` or a similarly named command.
- Process exactly one ticket per invocation.
- Require a clean working tree before starting.
- Run `/ticket-status` and `/ticket-doctor` before making changes.
- Use the existing ticket workflow commands as checkpoints where practical, including readiness, planning, activation check, verification, and completion brief.
- Perform ticket transitions and log updates only when checkpoint results are acceptable under the trusted mode rules.
- Launch implementation work through a bounded Pi process or equivalent orchestration mechanism.
- Stop on unresolved dependencies, failed or inconclusive checks, dirty unexpected state, workflow corruption, implementation failure, verification failure, or unclear advisory output.
- Commit a focused change only when implementation and verification pass.
- Report what was done, what commands were run, and where it stopped.

Out of scope:

- Bulk processing multiple tickets.
- Bypassing ticket workflow directories as the source of truth.
- Ignoring failed checks.
- Unlimited nested Pi processes.
- Autonomous repair of workflow corruption.
- Running without an explicit trusted or experimental command name.
- Replacing the manual workflow for all users.

## Acceptance Criteria

- [ ] The command is clearly marked experimental/trusted in documentation and output.
- [ ] The command refuses to start when the working tree is dirty.
- [ ] The command runs deterministic workflow checks before mutating ticket state.
- [ ] The command processes only one ticket and keeps ticket state transitions auditable through git.
- [ ] The command stops safely on failed, inconclusive, stale, or missing checkpoint results.
- [ ] The command records ticket log entries for transitions it performs.
- [ ] The command commits only after implementation and verification pass.
- [ ] The command reports a clear final status and next action.

## Dependencies

- `PLF-026` — the ticket doctor should exist first so the autopilot can run a deterministic workflow consistency check before trusting automation.

## Implementation Notes

Likely files:

- `.pi/extensions/ticket-workflow/`
- `docs/ticket-workflow-commands.md`
- possibly a new helper module or script if orchestration becomes too large for the extension entry point

Before moving this ticket to `planned/`, decide whether to split the work into smaller tickets:

1. a dry-run autopilot that only prints the intended sequence and stop conditions;
2. a trusted write-capable autopilot that performs transitions, implementation, verification, and commit.

A split may be safer, but the backlog ticket captures the desired end state: an explicitly trusted mode that can reduce manual ticket workflow steps when the user accepts the risk.

## Verification

- Run the command on controlled fixture tickets for success and failure paths.
- Confirm it refuses to start with a dirty working tree.
- Confirm it stops on failed doctor, failed activation check, failed verification, and unexpected changed files.
- Confirm a successful run leaves a focused commit and no active ticket.
- Confirm nested Pi process limits prevent unbounded recursion.
