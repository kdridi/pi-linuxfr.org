# PLF-005: Write run manifests for research runs

Backlog tickets may be incomplete. Before moving this ticket to `planned`, refine the scope and verification method.

## Objective

Create a lightweight run manifest that records seeds, selected candidates, collected details, failures, timestamps, and local paths for a bounded LinuxFr research run.

## Context

The expanded Agentic AI Career Radar review found that the local raw dataset is cumulative and not scoped to one run. This makes reports harder to audit and reproduce.

Motivating sources:

- `docs/linuxfr-v2-system-roadmap.md`
- `docs/prototype-prompts/agentic-ai-career-radar-expanded/review.md`
- `docs/prototype-prompts/agentic-ai-career-radar-expanded/result.md`

## Scope

In scope:

- Define a minimal manifest shape for one bounded run.
- Record seed URLs attempted, collection outcomes, selected detail URLs, local raw paths, cache/fresh information when available, failures, and timestamps.
- Keep the manifest local, human-readable or simple JSON/Markdown.

Out of scope:

- Scheduling runs.
- Global dataset indexing.
- Full provenance database.
- Automatically deciding what to collect.

## Acceptance Criteria

- [ ] To be refined before planning.
- [ ] A fresh agent can tell which sources belonged to one run.
- [ ] The manifest cites local paths and source URLs.
- [ ] The implementation does not replace raw sources or wiki notes.

## Dependencies

- None known.

## Implementation Notes

Possible locations to discuss during refinement:

- `data/runs/<run-id>/manifest.json`; or
- `data/wiki/runs/<run-id>.md`; or
- a small manifest file attached to generated reports.

Avoid designing a full experiment-tracking platform.
