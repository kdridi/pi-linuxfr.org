# PLF-036: Add trusted planned queue implementation runner

## Objective

Add a trusted queue runner that executes an approved implementation contract across multiple planned tickets in order, producing one focused commit per ticket and stopping at the first unsafe or failed step.

## Context

The user wants to review a global implementation contract once, approve it, then let Pi and child agents work for a long period on the planned ticket set. The runner should not re-plan the batch. It should consume the approved contract produced by `PLF-034` and call the single-ticket executor from `PLF-035` for each ticket in the approved order.

The goal is a long-running but auditable automation pass: each ticket runs in a bounded context, each successful ticket produces its own commit, and the queue stops on any failure or divergence.

## Scope

In scope:

- Add a trusted command such as `/ticket-implementation-run <contract-path-or-id> --trusted`.
- Require an approved implementation contract.
- Require a clean working tree before starting the queue.
- Run deterministic status and doctor checks before the queue and after each ticket.
- Execute tickets only in the order recorded in the approved contract.
- Delegate each ticket to the single-ticket executor from `PLF-035`.
- Preserve one focused commit per successful ticket.
- Stop at the first failed, inconclusive, stale, dirty, out-of-scope, or workflow-corrupt state.
- Produce a final queue report listing completed tickets, commits, skipped tickets, stop reason, and next human action.

Out of scope:

- Reordering tickets during execution.
- Re-planning tickets during execution.
- Continuing after a failed ticket.
- Bulk committing multiple tickets together.
- Applying backlog planning contracts.
- Running without explicit trusted approval.

## Acceptance Criteria

- [ ] The command refuses to run without an approved implementation contract.
- [ ] The command refuses dirty unexpected working trees.
- [ ] The command processes tickets in the contract order only.
- [ ] Each successful ticket results in one focused commit.
- [ ] The queue stops at the first failed or unsafe ticket and reports the reason.
- [ ] The queue does not re-run strategic planning for tickets already covered by the contract.
- [ ] The final report is sufficient for the human to resume or repair the queue later.
