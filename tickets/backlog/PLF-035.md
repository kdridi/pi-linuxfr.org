# PLF-035: Add trusted single-ticket implementation executor

## Objective

Add a trusted executor that can implement one ticket from an approved implementation contract section, verify it, complete it, and create a focused commit while stopping on divergence.

## Context

Before running a full planned queue, the system needs a reliable atomic execution unit. A single-ticket executor should consume the approved implementation contract produced by `PLF-034`, activate exactly one planned ticket, launch bounded write-capable implementation work, run the contract-listed verification commands, run workflow verification/checklist commands, move the ticket to completed, and commit if everything passes.

The executor must not redo strategic planning. It may use LLM coding agents for implementation, but they should receive the approved ticket-specific contract section as their task boundary.

## Scope

In scope:

- Add a trusted command or internal runner for one contract-listed ticket.
- Require a clean working tree before starting.
- Verify the approved contract fingerprints for the target ticket before mutation.
- Run deterministic status and doctor checks before mutation.
- Move the target ticket from `planned/` to `ongoing/` and record log entries when allowed by the contract.
- Launch a bounded write-capable Pi process or equivalent implementation worker with the approved ticket-specific plan.
- Run verification commands listed in the approved contract when practical.
- Run `/ticket-verify` and `/ticket-completion-brief` or their internal equivalents.
- Allow bounded repair attempts only when they stay inside the approved contract scope.
- Move the ticket to `completed/` and create a focused commit only when verification passes.
- Stop with a clear report on stale contract, dirty unexpected state, failed tests, failed verification, scope drift, or worker failure.

Out of scope:

- Processing more than one ticket per invocation.
- Choosing ticket order.
- Re-planning the ticket strategically.
- Applying backlog planning contracts.
- Ignoring failed checks.
- Continuing after a failed commit or workflow doctor failure.

## Acceptance Criteria

- [ ] The executor refuses to run without an approved implementation contract section for the target ticket.
- [ ] The executor refuses stale contracts and dirty unexpected working trees.
- [ ] The executor activates only the target ticket.
- [ ] The implementation worker receives the approved contract section as its scope boundary.
- [ ] Verification commands and workflow verification are run or clearly reported as unavailable.
- [ ] The executor stops on failure without moving to the next ticket.
- [ ] A successful run creates one focused commit and leaves no active ticket.
- [ ] The final report includes commands run, files changed, commit ID, and any warnings.
