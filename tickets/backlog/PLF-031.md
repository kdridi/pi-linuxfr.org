# PLF-031: Define approved automation contract model

## Objective

Define the durable contract model used by future trusted ticket automation commands so analysis happens once, the human approves the resulting contract, and execution applies that approved contract without redoing strategic analysis.

## Context

The manual ticket workflow is now operational, but the next automation layer should support two higher-level flows:

1. planning selected backlog tickets into a coherent `planned/` set, including dependency closure, ticket edits, and splits;
2. implementing planned tickets as a validated execution batch with per-ticket commits.

The key safety rule is not a traditional dry-run/re-run split. Instead, commands should build an approved automation contract from one analysis pass, then execute that exact contract after human approval. This avoids non-deterministic LLM re-analysis and avoids spending tokens twice on the same strategic analysis.

## Scope

In scope:

- Define automation contract artifact locations and metadata.
- Define contract states such as proposed, approved-by-human, running, completed, stopped, or obsolete if useful without making artifacts authoritative workflow state.
- Define required fingerprints for every source ticket and relevant project file used by the contract.
- Define how contracts record per-ticket child analysis results.
- Define the rule that execution must apply the approved contract and must not redo strategic planning.
- Define stop conditions when the repository no longer matches the approved contract.
- Document the distinction between read-only analysis children and trusted write-capable execution children.
- Document that ticket directories remain the workflow source of truth.

Out of scope:

- Implementing backlog planning commands.
- Implementing implementation batch commands.
- Adding write-capable child execution.
- Moving tickets or editing tickets automatically.
- Replacing manual workflow commands.

## Acceptance Criteria

- [ ] The approved automation contract concept is documented.
- [ ] Contract artifact paths and metadata are defined.
- [ ] The documentation requires source fingerprints sufficient to detect stale contracts before execution.
- [ ] The documentation states that execution consumes the approved contract instead of redoing strategic LLM analysis.
- [ ] The documentation defines stop conditions for stale inputs, unexpected dirty state, workflow doctor failures, and scope divergence.
- [ ] Ticket directories remain documented as authoritative workflow state.
