# PLF-018: Add deterministic ticket status command

## Objective

Add the first project-local Pi ticket workflow command, `/ticket-status`, to inspect the file-based ticket workflow without mutating repository files.

## Context

The ticket workflow is currently manual and repeated across Pi sessions. Challenge responses for `docs/use-case-challenge-prompts/sub-pi-ticket-workflow-orchestration/` converged on starting with deterministic read-only checks before adding child Pi orchestration.

This ticket is the smallest useful foundation: it proves a ticket workflow extension can load, gives immediate visibility into ticket state, and encodes the critical invariant that at most one ticket may be in `tickets/ongoing/`.

## Scope

In scope:

- Add a project-local Pi extension for ticket workflow commands, likely under `.pi/extensions/ticket-workflow/`.
- Register `/ticket-status`.
- Count `PLF-*.md` tickets in `backlog/`, `planned/`, `ongoing/`, `completed/`, and `rejected/`.
- Report the current ongoing ticket when exactly one exists.
- Report an error or warning when more than one ongoing ticket exists.
- Report missing expected ticket state directories.
- Keep the command read-only.

Out of scope:

- Child Pi sessions.
- Advisory artifact generation.
- Ticket state transitions.
- Ticket editing.
- Commit automation.
- Deep ticket validation beyond simple state counts and ongoing invariant checks.

## Acceptance Criteria

- [ ] `/ticket-status` is available after Pi reloads project-local extensions.
- [ ] `/ticket-status` reports counts for backlog, planned, ongoing, completed, and rejected tickets.
- [ ] It reports that no ticket is active when `tickets/ongoing/` contains no `PLF-*.md` file.
- [ ] It reports the active ticket ID when exactly one ongoing ticket exists.
- [ ] It reports a clear workflow error when more than one ongoing ticket exists.
- [ ] It does not modify any repository file.
- [ ] The command output states or implies that ticket directories remain the source of truth.

## Dependencies

- None.

## Implementation Notes

Likely files:

- `.pi/extensions/ticket-workflow/index.ts`
- `.pi/extensions/ticket-workflow/package.json` if needed
- `docs/ticket-workflow-commands.md` if command documentation is added now

Prefer a small implementation. Start single-file if that is easier to inspect; split helpers only when later tickets need it.

## Verification

- Reload Pi or start a fresh Pi session and confirm `/ticket-status` is registered.
- Run `/ticket-status` in the current repository state and confirm it reports the expected counts and no active ongoing ticket.
- Manually inspect `git status --short` before and after running the command to confirm no files were modified by the command.
- If practical, use temporary controlled ticket files to confirm the one-ongoing and multiple-ongoing reporting behavior, then remove the temporary files.
