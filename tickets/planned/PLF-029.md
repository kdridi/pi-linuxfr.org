# PLF-029: Define parent LLM handoff for advisory ticket commands

## Objective

Define and implement the minimal parent LLM handoff convention for future advisory ticket workflow commands.

## Context

`PLF-020` added a read-only child Pi advisory runner, and `PLF-028` made `/ticket-child-diagnostic` visibly acknowledge child process startup. Manual UX review showed another important gap: command output displayed as custom command messages is useful to humans, but it is not automatically reworked by the parent LLM as a normal tool result would be.

The advisory command family should not only display child output. When a command produces a verdict, plan, verification result, or recommendation, it should hand the advisory result back to the parent LLM so the main conversation can synthesize it, explain what matters, and ask for human validation before any state transition.

## Scope

In scope:

- Document the parent LLM handoff rule for advisory ticket commands.
- Define when advisory commands should use `sendUserMessage(...)` to trigger a parent LLM follow-up.
- Define what the handoff message must contain: original request, artifact path when present, advisory summary or child output, advisory-only notice, and explicit instruction not to mutate ticket state without human approval.
- Define print/non-interactive behavior: do not trigger a parent LLM follow-up; instead print a suggested handoff message.
- Add a small shared helper in the ticket workflow extension for building or sending advisory parent handoff messages if it keeps later commands consistent.
- Update backlog tickets for future advisory commands so they depend on this convention where appropriate.

Out of scope:

- Implementing `/ticket-readiness`, `/ticket-plan`, `/ticket-verify`, `/ticket-activate-check`, `/ticket-completion-brief`, or `/ticket-doctor`.
- Changing the read-only child Pi runner safety boundary.
- Adding child output streaming.
- Automatically moving tickets, committing changes, or approving workflow transitions.
- Making `/ticket-status` trigger a parent LLM follow-up.

## Acceptance Criteria

- [ ] `docs/ticket-workflow-commands.md` documents the advisory parent handoff pattern.
- [ ] The documentation distinguishes deterministic display-only commands from advisory commands that should trigger a parent LLM follow-up.
- [ ] The convention says advisory commands should use parent LLM handoff for verdicts, plans, verification results, and transition recommendations.
- [ ] The convention says print/non-interactive mode must not trigger a hidden LLM follow-up and should print a suggested handoff instead.
- [ ] A shared helper or clearly documented message template exists for future advisory commands.
- [ ] Tickets `PLF-021` through `PLF-025` explicitly account for the parent handoff convention.
- [ ] `PLF-026` remains documented as deterministic/read-only without parent handoff by default.
- [ ] TypeScript extension checking passes if extension code changes.

## Implementation Notes

Likely files:

- `.pi/extensions/ticket-workflow/index.ts` if a shared helper is added.
- `docs/ticket-workflow-commands.md`.
- `tickets/backlog/PLF-021.md` through `tickets/backlog/PLF-026.md` if additional dependency or scope clarifications are needed.
- `tickets/completed/PLF-029.md` when completed.

A minimal helper can build a Markdown handoff message and optionally call `pi.sendUserMessage(...)` only when the command is not running in print mode. Keep the helper small; later command tickets should decide exactly when to call it.

Suggested handoff content:

```markdown
A ticket workflow advisory command completed.

Original request:
<request>

Advisory artifact:
<path or none>

Advisory result:
<summary or child output>

This is advisory only. Ticket files and ticket directories remain authoritative.
Please synthesize the result for the user, explain what matters, propose the next action, and do not mutate ticket state without explicit human approval.
```
