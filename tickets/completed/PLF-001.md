# PLF-001: Add bounded candidate extraction from collected seed pages

## Objective

Add a small, deterministic way to extract candidate LinuxFr detail URLs from already collected seed pages so agents can build inspectable shortlists instead of selecting detail pages manually.

## Context

The Agentic AI Career Radar expanded review found that the current MVP works for collect-inspect-synthesize loops, but candidate selection is still manual and hard to reproduce. The V2 roadmap identifies candidate extraction from feeds, tags, and listings as the highest-priority next capability.

This ticket should improve repeatability without becoming a crawler. It must operate only on local raw sources that have already been collected.

## Scope

In scope:

- Extract LinuxFr internal detail URLs from selected local raw source files.
- Deduplicate candidates by canonical URL or path.
- Classify obvious candidate types such as news, journal, link, forum, poll, or unknown.
- Return inspectable fields such as candidate URL, source path, source URL, title or anchor text when available, snippet when available, matched terms when provided, and a simple score or reason.
- Keep the implementation bounded and local-only.

Out of scope:

- Crawling LinuxFr.
- Collecting candidate detail pages automatically.
- Collecting comments.
- Building a vector index or semantic search system.
- Producing final reports or wiki notes.

## Acceptance Criteria

- [x] The feature works from existing files under `data/raw/` and does not fetch new pages.
- [x] The output includes enough source context for an agent to decide which candidate URLs to collect next.
- [x] Candidate URLs are deduplicated and classified when possible.
- [x] The implementation remains compatible with the bounded local MVP approach.
- [x] Basic verification covers at least one collected feed, tag, listing, or HTML/Markdown seed source.

## Dependencies

- None.

## Implementation Notes

Likely implementation options:

- Extend `linuxfr_query_raw` with a candidate-extraction mode; or
- Add a separate narrowly scoped Pi tool if that keeps the existing query tool simpler.

Prefer the smallest implementation after inspecting the current extension code and tests.

## Verification

- Checked for existing package scripts: none are defined for `.pi/extensions/linuxfr-dataset`.
- Manual runtime check with Node's TypeScript stripping imported the extension through a temporary copy with a `typebox` stub and confirmed that the registered tool list includes `linuxfr_extract_candidates`.
- Manual runtime check executed `linuxfr_extract_candidates` through the temporary extension copy against `data/raw/pages/feed-a554f7e7d2.atom` and `data/raw/pages/tag-07e4f4520b.html` with `query: "agent llm"` and `limit: 5`; result: 2 local sources read, 47 candidate occurrences found, 5 bounded candidates returned, no network fetch performed.
- Manual helper checks covered an Atom feed, a tag HTML page, and a listing HTML page; candidates were deduplicated and classified as LinuxFr detail types.

## Decisions

- Added a dedicated `linuxfr_extract_candidates` tool instead of extending `linuxfr_query_raw`, because candidate extraction has a separate single responsibility and clearer tool-selection semantics for LLM use.
- Kept extraction local-only: the tool reads collected metadata and files under `data/raw/` and does not fetch or collect candidate detail pages.
- Default extraction uses recent local seed-like sources (`feed`, `tag`, and `listing`) when no explicit source path or URL is provided, while remaining bounded by `maxSources` and `limit`.
- Scoring is intentionally simple and inspectable: classified detail type, title/anchor presence, snippet/date context, and optional query-term matches.
