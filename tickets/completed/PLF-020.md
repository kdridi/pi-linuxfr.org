# PLF-020: Add read-only child Pi advisory runner

## Objective

Implement a reusable helper for ticket workflow commands to run bounded read-only child Pi sessions and capture their advisory Markdown output.

## Context

The desired workflow keeps the main Pi session as the human-facing orchestrator while delegating context-heavy analysis to child Pi sessions. Before implementing specific advisory commands, the child-runner primitive must be small, safe, and easy to inspect.

## Scope

In scope:

- Add helper code that spawns a child Pi process for advisory analysis.
- Restrict child sessions to read-only tools: `read`, `grep`, `find`, and `ls` only.
- Exclude `write`, `edit`, and `bash` from the child tool allowlist.
- Disable child extensions, prompt templates, skills, and themes where supported.
- Use ephemeral child sessions with `--no-session`.
- Add a recursion guard environment variable, such as `PI_TICKET_CHILD=1`.
- Capture child Markdown output, stderr, exit status, and child failure status.
- Surface clear errors to the orchestrator when the child run fails.
- Keep the parent command responsible for any artifact writes.
- Add a small read-only diagnostic command only if needed to verify the helper.

Out of scope:

- Write-capable child sessions.
- Specific readiness, planning, or verification semantics.
- Parallel child sessions.
- Long-running background processes.
- Automatic ticket movement or commits.

## Acceptance Criteria

- [x] A helper or diagnostic path can invoke a child Pi process and capture Markdown output.
- [x] Child Pi sessions cannot use `write`, `edit`, or unrestricted `bash` through the configured tool allowlist.
- [x] Child Pi sessions do not load the ticket workflow extension recursively.
- [x] Child failures are reported clearly to the parent command.
- [x] No repository files are modified by the child runner itself.

## Dependencies

- `PLF-018` — command extension foundation.
- `PLF-019` — artifact conventions for later commands using this runner.

## Implementation Notes

Likely files:

- `.pi/extensions/ticket-workflow/`
- `docs/ticket-workflow-commands.md`

Prefer the simplest subprocess interface first, likely `pi -p`, unless JSON mode proves necessary for reliable output capture. Keep the command line documented because this is a safety boundary.

The intended first command line is:

```bash
pi --approve --no-session --no-extensions --no-skills --no-prompt-templates --no-themes --tools read,grep,find,ls -p '<prompt>'
```

The child environment should include `PI_TICKET_CHILD=1`, and parent commands should refuse to spawn another child when that variable is already set.

## Decisions

- Implemented a reusable exported `runReadOnlyChildPiAdvisory(...)` helper in the existing ticket workflow extension.
- Added `/ticket-child-diagnostic` only as a minimal verification path for the runner primitive.
- Used `pi --approve --no-session --no-extensions --no-skills --no-prompt-templates --no-themes --tools read,grep,find,ls -p '<prompt>'` as the child safety boundary.
- Excluded `bash` from the first read-only child allowlist to keep the mutation boundary simpler than a read-only-bash policy.
- Kept artifact writing out of this ticket; future parent commands remain responsible for advisory artifacts.
