# PLF-028: Add immediate progress feedback for child Pi diagnostics

## Objective

Make `/ticket-child-diagnostic` visibly acknowledge that a read-only child Pi advisory run has started before waiting for the child process to complete.

## Context

`PLF-020` added the reusable read-only child Pi advisory runner and a diagnostic command. The command works, but the current user experience is confusing because the parent command displays nothing while the child Pi process is running. A user can reasonably think the command did not start.

This ticket should fix only the immediate feedback gap before future advisory commands reuse the same pattern.

## Scope

In scope:

- Update `/ticket-child-diagnostic` to emit a visible starting/progress message before awaiting the child Pi process.
- Include the relevant safety boundary in the starting message: read-only tools and disabled child resources.
- Keep the existing final diagnostic output.
- Keep the existing child runner behavior and command-line safety boundary.
- Support both interactive/non-print modes and print mode.

Out of scope:

- Streaming child Pi output.
- Switching the child runner to JSON mode.
- Adding new advisory artifact commands.
- Changing ticket workflow state transitions.
- Changing the child tool allowlist.

## Acceptance Criteria

- [x] `/ticket-child-diagnostic` visibly reports that the child run is starting before the child process completes.
- [x] The starting message mentions the read-only child boundary.
- [x] The final child diagnostic output is still displayed after completion.
- [x] No repository files are modified by running the diagnostic command.
- [x] TypeScript extension checking passes.

## Dependencies

- `PLF-020` — the child diagnostic command and reusable read-only runner must exist first.

## Implementation Notes

Likely file:

- `.pi/extensions/ticket-workflow/index.ts`

A minimal implementation can add a small `renderChildDiagnosticStarting()` helper and call it before `await runReadOnlyChildPiAdvisory(...)`. In print mode, write the starting text to stdout immediately. In other modes, use `pi.sendMessage(...)` before awaiting the child.

Do not implement streaming in this ticket.
