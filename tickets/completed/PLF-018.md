# PLF-018: Add deterministic ticket status command

## Objective

Add the first project-local Pi ticket workflow command, `/ticket-status`, to inspect the file-based ticket workflow without mutating repository files.

## Context

The ticket workflow is currently manual and repeated across Pi sessions. Challenge responses for `docs/use-case-challenge-prompts/sub-pi-ticket-workflow-orchestration/` converged on starting with deterministic read-only checks before adding child Pi orchestration.

This ticket is the smallest useful foundation: it proves a ticket workflow extension can load, gives immediate visibility into ticket state, and encodes the critical invariant that at most one ticket may be in `tickets/ongoing/`.

## Scope

In scope:

- Add a project-local Pi extension for ticket workflow commands, likely under `.pi/extensions/ticket-workflow/`.
- Register `/ticket-status`.
- Count `PLF-*.md` tickets in `backlog/`, `planned/`, `ongoing/`, `completed/`, and `rejected/`.
- Report the current ongoing ticket when exactly one exists.
- Report an error or warning when more than one ongoing ticket exists.
- Report missing expected ticket state directories.
- Keep the command read-only.

Out of scope:

- Child Pi sessions.
- Advisory artifact generation.
- Ticket state transitions.
- Ticket editing.
- Commit automation.
- Deep ticket validation beyond simple state counts and ongoing invariant checks.

## Acceptance Criteria

- [x] `/ticket-status` is available after Pi reloads project-local extensions.
- [x] `/ticket-status` reports counts for backlog, planned, ongoing, completed, and rejected tickets.
- [x] It reports that no ticket is active when `tickets/ongoing/` contains no `PLF-*.md` file.
- [x] It reports the active ticket ID when exactly one ongoing ticket exists.
- [x] It reports a clear workflow error when more than one ongoing ticket exists.
- [x] It does not modify any repository file.
- [x] The command output states or implies that ticket directories remain the source of truth.

## Verification

- `pi --approve --no-session --mode json -p '/ticket-status' | rg '"customType":"ticket-status"|Active ticket'` confirmed the command is registered and emits a `ticket-status` custom message.
- `pi --approve --no-session -p '/ticket-status'` reported counts for the current repository while `PLF-018` was active: backlog 24, planned 0, ongoing 1, completed 1, rejected 0.
- Manual before/after `git status --short` comparison around `pi --approve --no-session -p '/ticket-status'` confirmed the command did not modify repository files.
- Temporary controlled directories run with `pi --no-extensions -e .pi/extensions/ticket-workflow/index.ts --no-session -p '/ticket-status'` confirmed no-ongoing, one-ongoing, multiple-ongoing, and missing-directory reporting. Temporary files were removed.
- Final current-state check after moving the ticket to completed with `pi --approve --no-session -p '/ticket-status'` reported backlog 24, planned 0, ongoing 0, completed 2, rejected 0, and no active ticket.

## Files Changed

## Decisions

- Implemented `/ticket-status` as a project-local Pi command, not an LLM-callable tool, because the primary consumer is the human/orchestrator workflow.
- Kept the command read-only: it only reads ticket directories and emits status output.
- Kept validation shallow and deterministic: state counts, missing state directories, and the ongoing-ticket invariant only.
- Added print-mode output for non-interactive verification while using a custom displayed message for TUI/RPC/JSON modes.

## Notes

This is intentionally not a child-agent ticket. The deterministic foundation should exist before advisory child Pi orchestration is added.

## Resolution

Implemented as a focused project-local Pi extension command. The final commit message includes `PLF-018`; the commit identifier is discoverable with `git log --grep='PLF-018'`.
