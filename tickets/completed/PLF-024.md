# PLF-024: Add planned ticket activation check command

## Objective

Add a read-only helper command that checks whether a planned ticket is ready to move to `tickets/ongoing/` and prints the safe manual transition steps.

## Context

The ticket workflow requires exactly one ongoing ticket. Activation is a meaningful state transition and should remain human-approved. A command can still reduce repeated checks by verifying preconditions and showing the exact next steps.

## Scope

In scope:

- Add a command such as `/ticket-activate-check <ticket-id>`.
- Require the target ticket to exist in `tickets/planned/`.
- Require `tickets/ongoing/` to contain no `PLF-*.md` file.
- Warn if an implementation plan artifact is missing or stale.
- Check explicit dependencies enough to warn about unresolved items.
- Print suggested manual commands or steps, including `git mv`, without executing them.
- Explain that human approval is required before activation.
- Hand the activation recommendation back to the parent LLM according to the `PLF-029` convention when the result needs explanation or human decision support.

Out of scope:

- Moving the ticket automatically.
- Editing ticket logs automatically.
- Overriding unresolved dependencies.
- Starting implementation.

## Acceptance Criteria

- [x] The command passes only when the ticket is planned and `ongoing/` is empty.
- [x] The command fails clearly when another ticket is already ongoing.
- [x] The command warns when the implementation plan is missing or stale.
- [x] The command prints exact suggested manual transition steps.
- [x] The command hands the activation recommendation back to the parent LLM or prints the suggested handoff in print mode.
- [x] The command does not mutate files.

## Decisions

- Implemented `/ticket-activate-check <ticket-id>` as a read-only advisory command that only accepts planned tickets and requires `tickets/ongoing/` to be empty.
- Kept the command deterministic and parent-side only; it does not spawn a child Pi process because the activation check is a bounded workflow precondition check.
- Did not write an advisory artifact because the ticket explicitly requires the command not to mutate files; parent handoff uses `(none)` for the artifact path.
- Reused the existing dependency and plan-artifact inspection helpers so unresolved dependencies and missing or stale implementation plans are visible before activation.
- Printed manual activation steps, including the `git mv` command, while keeping human approval mandatory.
