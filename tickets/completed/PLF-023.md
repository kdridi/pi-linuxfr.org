# PLF-023: Add ongoing ticket verification brief command

## Objective

Add a read-only advisory command that verifies the single ongoing ticket against its acceptance criteria, implementation plan, and current repository changes.

## Context

Completing a ticket safely requires checking scope, acceptance criteria, changed files, and verification evidence. A child Pi can perform this review in a separate context and produce a verification brief, while the orchestrator and human decide whether completion is appropriate.

## Scope

In scope:

- Add a command such as `/ticket-verify` for the current ongoing ticket.
- Require exactly one `PLF-*.md` file in `tickets/ongoing/`.
- Read the ongoing ticket.
- Read the latest non-stale implementation plan artifact when present.
- Inspect changed files or git diff enough to summarize implementation scope.
- Use read-only child Pi analysis.
- Write an advisory verification brief to the agreed artifact location.
- Include acceptance criteria checklist, scope review, changed files, verification evidence, and a verdict such as `pass`, `fail`, or `inconclusive`.
- Hand the advisory verification result back to the parent LLM according to the `PLF-029` convention.

Out of scope:

- Completing the ticket.
- Committing changes.
- Moving files.
- Running destructive commands.
- Fixing failed verification automatically.

## Acceptance Criteria

- [x] The command refuses to run when there is zero or more than one ongoing ticket.
- [x] The verification brief includes an advisory notice and source ticket fingerprint.
- [x] The brief references a plan artifact when one exists and warns if it is stale.
- [x] The brief lists changed files or a changed-files snapshot.
- [x] The brief marks unresolved acceptance criteria as failed or inconclusive, not silently passed.
- [x] The brief recommends whether completion is ready for human decision without performing completion.
- [x] The command hands the verification result back to the parent LLM or prints the suggested handoff in print mode.

## Dependencies

- `PLF-020` — read-only child Pi runner.
- `PLF-022` — implementation plan artifacts are useful context for verification.
- `PLF-029` — parent LLM handoff convention for advisory command results.

## Decisions

- Implemented `/ticket-verify` as a read-only advisory command requiring exactly one `PLF-*.md` file in `tickets/ongoing/`.
- Added bounded parent-side git inspection via `execFileSync("git", ["status", "--porcelain"])` to summarize changed files without dumping large diffs.
- Added `inspectPlanArtifact(...)` to read `tickets/.artifacts/plans/<id>.md` when present and detect staleness by comparing the recorded `ticketSha256` frontmatter to the current ticket SHA-256.
- Reused the existing read-only child Pi runner safety boundary; the parent command writes the advisory artifact.
- Reused existing advisory artifact metadata and parent handoff helpers so verification briefs follow the same conventions as readiness and plan artifacts.
- Used an inline prompt helper rather than adding a separate prompt file to keep the MVP implementation small.
